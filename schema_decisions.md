# 스키마 설계 결정 로그 (Schema Design Decision Log)
> 목적: template.json, asset_db.json, edit_instruction.json 확정 전 논의된 모든 설계 결정과 근거를 기록 
> 주의: 본 문서의 [RULE] 항목은 Claude Code가 파이프라인 개발 시 절대적으로 준수해야 하는 강제 지시문임.

---

## [RULE 1] 트랜지션 처리: 하드컷 강제

- **규칙:** template.json에는 트랜지션 필드를 생성하지 않으며, 렌더링 파이프라인(Pink)의 모든 컷 전환은 **하드컷**으로 강제한다.
- **사유:** 트랜지션 타입(디졸브, 휩, 줌 등)을 식별하는 것은 현재 기술(PySceneDetect) 능력 밖이며, 별도의 비전 모델 학습 및 추론의 영역이므로 MVP 아키텍처에서는 배제한다. 숏폼 문법의 99%는 컷편집 중심이므로 하드컷만으로도 성립한다.

---

## [RULE 2] 자막(Subtitle) 처리: 텍스트, 위치, 타이밍, 크기 변동 기록

- **규칙:** template.json에 원본 자막의 **텍스트 내용 + 위치 + 타이밍(in/out)** 뿐만 아니라, 숏폼에서 리듬감을 타격하는 **글자 크기 변동(Scale/Highlight)** 속성도 함께 분류하여 기록한다.
- **사유:** 글자가 확 커지는 팝업 효과 등은 단순한 그래픽이 아니라 영상의 리듬(비트)을 시각적으로 구성하는 핵심 요소이므로 배제해선 안 된다. 폰트 저작권 이슈는 상업용 무료 폰트(OFL - G마켓 산스 등)를 시스템에 강제 내장하는 방식으로 합법적으로 해결할 수 있다.

```json
/* 적용 예시 */
"subtitle": {
  "text": "서울 3대 국밥",
  "position": "center",
  "in": 0.0,
  "out": 1.5,
  "style": {
    "is_highlighted": true,  /* 크기 변동, 색상 강조 존재 여부 */
    "animation": "pop_up"    /* 식별된 애니메이션 텍스트 */
  }
}
```

## [RULE 3] 오디오 분석 깊이: BPM 및 비트 배열 추출 한정

- **규칙:** Yellow 파이프라인 오디오 분석 시 librosa를 통해 `beat_timestamps` 배열(ms 단위)과 `BPM` 수치만 추출한다. 나중에 Agent B는 이 비트 배열만을 기준으로 컷 전환 엇박자(±100ms)를 검증한다.
- **사유:** MVP 대상인 정적 영상(음식점 홍보, 먹방 등)은 BGM의 비트가 명확한 바이럴 숏폼 음악을 쓰므로 이 데이터만으로 컷 포인트 판별이 충분함. 불필요한 연산 제거를 통한 VLM/인프라 비용 방어 목적.
- **예외/확장:** 게임 도메인(타격음/총소리 위주) 확장 시 오디오 최우선 파이프라인의 일환으로 `energy_curve`(소리 크기 변화) 및 `onset_strength`(소리 시작점) 데이터 추출을 추가한다.

## [RULE 4] 컷 순서 재배열 절대 금지

- **규칙:** Agent A는 template.json에 정의된 컷의 시간적 배열 순서를 100% 준수하여 타임라인 빈칸을 채워야 한다. 유저 에셋(Blue)의 매칭 점수를 인위적으로 높이기 위해 원본 템플릿의 컷 순서를 뒤섞어서는 안 된다.
- **사유:** "유행하는 편집 문법(인트로→전경→클로즈업 등)을 발췌하여 그대로 복제한다"는 프로젝트 핵심 철학을 유지하기 위함. 순서 변경 시 숏폼 문법이 파괴되며, 무엇보다 단방향 슬롯 배열로 제한함으로써 Agent A의 매칭 연산 복잡도와 레이턴시를 획기적으로 낮출 수 있다.

---

## [RULE 5] Blue 세그먼트 분할 상한선 강제 (검토 중)

- **규칙:** 브라우저(클라이언트) 단에서 Optical Flow로 원본 클립을 분할할 때, 1개의 원본 영상당 **최대 5개**의 세그먼트까지만 분할(프레임 간 픽셀 변화량 절대치가 가장 컸던 TOP 5 지점 기준)을 허용한다. 또한 각 세그먼트의 **최소 길이는 1.0초** 이상이어야 한다.
- **사유:** 무제한 세그먼트 분할을 허용할 경우 1개 영상에서 수십 번의 VLM API 호출이 발생하여 운영 비용이 폭증함. 상한선을 두어 전체 파이프라인의 최대 API 과금 구조를 완벽히 통제 가능하게 함.

---

## [RULE 6] Semantic 매칭: 임베딩 벡터 기반 코사인 유사도

- **규칙:** Agent A가 템플릿의 요구 시각 속성(텍스트)과 유저 에셋의 텍스트 태그 간의 유사도(Semantic Score)를 채점할 때, LLM 프롬프팅 방식이 아닌 **경량 텍스트 임베딩 모델(`all-MiniLM-L6-v2`)을 통한 코사인 유사도(Cosine Similarity) 연산**을 수행한다.
- **사유:** VLM이 추출한 자연어 풀이를 비용과 레이턴시가 막대한 LLM API로 매칭하는 것은 아키텍처적 낭비임. 임베딩 벡터 수학 연산은 "Boiling"과 "Stewing" 같은 유의어도 완벽히 잡아내는(품질 타협 없음) 최고 효율의 매칭 방식이기 때문.

---

## [RULE 7] 비정상 소스 분량 미달 방어 (Fail Fast)

- **규칙:** 사용자가 업로드한 원본 영상(Blue)들의 총합이 템플릿의 최소 런타임(예: 20초)조차 채우지 못하는 극단적 비정상(Abnormal) 케이스일 때만 파이프라인 진입을 차단하고 **즉각 에러 반환(Reject)**한다.
- **사유:** 실제 환경에서 사용자는 숏폼 하나를 위해 5분~20분 단위의 충분한 원본을 업로드하므로 소스 부족 사태는 정상적인 루트에선 발생하지 않음. 단, 시스템 방어 차원에서 찌꺼기 영상(5초 미만)만 던지는 기조작 봇/어뷰징 케이스를 차단하기 위한 최소한의 입구 컷 장치임.

---

## [RULE 8] 원본 흐름 보존 및 하이브리드 속도 변조 — 모드별 분기

- **규칙:** 유저가 선택한 편집 방식(비트 우선 / 서사 우선)에 따라 Agent A의 타임라인 연산과 속도 변조 전략이 달라진다. Agent A는 최종 `speed` 배수와 `timeline_position`을 모두 계산하여 edit_instruction.json에 기록하며, Pink는 지시서대로만 실행한다(Dumb 렌더러).

**[비트 우선 모드] 템플릿 슬롯 절대 고정:**
  1. **[1순위] Smart Trim:** 유저 소스의 핵심 구간만 발췌하여 슬롯 길이에 맞춤.
  2. **[2순위] Speed Ramp:** Trim으로도 부족할 경우 배속(예: `"speed": 2.0`)을 걸어 구겨 넣음. 하위 슬롯 타임라인 재계산 불필요.

**[서사 우선 모드] 타임라인 가변 허용:**
  1. **[1순위] 가변 슬롯 (1배속 유지):** 슬롯 자체를 유저 소스 길이에 맞춰 늘림. 이후 모든 슬롯의 `timeline_position`과 비트 기준점을 뒤로 미루는 **연쇄 지연(Cascading Shift) 재계산**을 Agent A가 수행.
  2. **[2순위] Speed Ramp:** Cascading Shift가 과도하게 누적될 경우(특정 Threshold 초과)에 한해 배속 압축으로 전환.

## [RULE 9] 현장음 선택적 보존 (Semantic Muting)

- **규칙:** 영상의 모든 현장음을 무조건 살리거나 무조건 죽이는 것이 아니라, **템플릿(Yellow)의 각 슬롯이 요구하는 시각적/맥락적 성격에 따라 오디오 삽입 여부를 분기 처리(true/false)**한다.
- **사유:** 고기 굽기, 한 입 먹기 등 음식(동작)이 주인공인 컷은 지글지글/바삭하는 현장음(ASMR)이 숏폼의 몰입도를 극대화하므로 현장음을 살려야 한다. 반면, 식당 외경이나 단순 간판 컷 등은 자동차 소음이나 불필요한 바람 소리만 들어갈 확률이 높으므로 무음 처리하는 것이 맞다.
- **구현 방식:** `template.json`의 각 컷 정보에 `"keep_audio": true/false` 플래그를 추가한다. Yellow 파이프라인에서 moondream2가 추출한 `objects` 태그(예: "boiling", "eating", "grilling" 등)를 규칙 기반으로 해석하여 이 값을 세팅한다. VLM이 오디오를 직접 분석하는 것이 아니라 시각 태그를 기준으로 추론한다. FFmpeg.wasm은 이 플래그를 보고 해당 컷의 오디오 트랙을 믹싱할지 Drop할지 결정한다.

## [RULE 10] 세그먼트 중복 할당: Soft Penalty (감점) 적용

- **규칙:** Agent A는 한 번 타임라인 매칭에 사용된 유저 클립(세그먼트)을 후보군에서 영구 삭제하지 않는다. 단, 동일 클립이 다음 슬롯 경쟁에 나설 때는 최종 스코어에 **30%의 감점(Soft Penalty)**을 부여한 상태로 다른 후보들과 경쟁시킨다.
- **사유:** 소스가 부족할 때 "한 번 쓴 건 절대 못 쓴다"는 규칙을 강제하면 후반부 슬롯에 전혀 내용이 맞지 않는 쓰레기 컷이 억지로 욱여넣어짐. 차라리 아주 잘 찍힌 매력적인 소스라면 30% 감점을 시켰음에도 1등을 차지하게 냅두어, 숏폼 내에서 해당 장면이 한 번 더 반복(수미상관/리플레이)되게 하는 것이 영상 퀄리티 방어에 훨씬 유리함.

## [RULE 11] Fallback 기준점: 구역별 UX 슬라이더 분리 적용 및 모델 매핑 보정

- **규칙:** Agent A가 강제 할당(Fallback)으로 넘어가는 기준점(임계값)을 영상 전체에 일괄 적용되는 단독 슬라이더로 두지 않는다. 최종적으로 **사용자가 프론트엔드 UI에서 '구간별(Intro, Body, Outro 등 슬롯 단위)'로 매칭 정확도를 따로 조절할 수 있는 개별 슬라이더** 형태로 개방한다.
- **사유:** 전체 영상에 단일 엄격도(예: 90%)를 들이밀 경우, 단 1개의 슬롯에서만 컷이 모자라도 전체 영상의 렌더링 파이프라인이 멈춰버리는(영상 자체를 못 뽑는) 치명적인 결함이 발생한다. 구역별로 정확도 허들을 다르게 주어야 비용/대기시간 방어와 무중단 렌더링이 성립한다.
- **개발 단계 수행 지침 [Action Item]:** 
  - 개발 단계(Mock Data Test)에서는 채택한 임베딩 모델의 실제 코사인 점수 분포를 추출한다.
  - 이를 바탕으로 사용자가 조작하는 프론트엔드 슬라이더의 **0~100 수치가 실제 모델의 어느 코사인 Range(예: 0.2 ~ 0.8)에 수학적으로 대응할지** 매핑 보정 작업을 반드시 선행한다.

## [RULE 12] Agent A/B 탐색 알고리즘 (루프 상한): 1-Hit Fallback 직행

- **규칙:** Agent A가 도출한 최고 점수(Top 1 Semantic) 타임라인 조합을 Agent B가 검수했을 때 반려(Reject)될 경우, **두 번째 기회를 주지 않고 즉시 시각(Semantic) 매칭을 포기하며 오디오 리듬(Motion) 강제 할당(순수 Fallback 모드)으로 폭포수처럼 넘어간다.** (핑퐁 루프 최대 1회 제한)
- **모드별 Agent B 검증 기준:** [비트 우선 모드]는 ±100ms 엇박자를 엄격 검증하여 Reject 가능. [서사 우선 모드]는 ±100ms 검증을 강제 Bypass하고 타임라인 총합 길이 부족 여부만 확인하므로, 실질적으로 Reject 발생 빈도가 낮음.
- **사유:** 연산적(CS)으로 접근했을 때, 가장 점수가 높은 1순위 조합이 오디오 비트를 못 맞춘다면 2~3순위 조합들 역시 비트를 맞출 확률이 희박하다. 의미 없는 재귀(Recursion) 루프를 돌며 핑퐁을 3~5회 반복하는 것은 서버 컴퓨팅 리소스와 사용자 대기 시간(레이턴시)을 심각하게 낭비하는 안티 패턴이기 때문이다. 한 번에 안 되면 바로 Fallback으로 내리는 것이 시스템 안정성에 기여한다.
- **추후 재논의 조건 [Action Item]:**
  - 기획이 고도화되어 1-Hit Fallback 로직의 품질 저하가 너무 눈에 띈다고 판단될 경우에 한하여, '통합 단일 스코어링 원패스 탐색(Option 1)'이나 'Top-K 병렬 검증(Option 2)' 같은 다른 알고리즘 튜닝안을 재고려(Reconsider)한다.

## [RULE 13] Per-slot 모드 선택 및 싱크 파괴 경고 플래그

- **규칙:** 슬롯별 편집 방식([비트 우선] / [서사 우선])은 개별 설정 가능하다. Agent A는 타임라인 연산 시 슬롯별 모드를 읽어 처리하며, 단 하나의 슬롯이라도 [서사 우선]으로 인한 Cascading Shift가 발생한 경우 edit_instruction.json 최상단에 `"is_original_sync_broken": true`를 강제 기록한다.
- **사유:** Cascading Shift는 시각적 비트 스냅만 복구할 뿐, 음악의 하이라이트(드랍) 위치는 복구하지 못한다. 사용자가 틱톡/인스타에서 유행 음원을 씌울 경우 하이라이트 싱크가 어긋나는 현상은 물리적으로 불가피하다. 이를 엔진 버그로 오인하지 않도록 시스템이 선제적으로 고지 책임을 진다.
- **Pink UX 처리:** `is_original_sync_broken: true`인 결과물은 렌더링 완료 화면 다운로드 버튼 상단에 하이라이트 싱크 불일치 경고를 강제 노출한다.
