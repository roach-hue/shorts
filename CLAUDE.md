# CLAUDE.md — 작업 규칙 (절대 준수)

> 이 규칙들은 이 레포의 **모든** 작업을 지배한다. 작업 진입 전 반드시 확인한다.
> 사용자가 반복적으로 명시한 강제 규칙이며, 어기는 것은 작업 거부 사유로 취급한다.

## 작업 규칙 (RULES OF ENGAGEMENT)

1. **병렬개발 절대 금지.** 한 번에 단 하나의 작업만. **단일 작업을 완수(테스트 green + 검증 통과)한 뒤**에야 다음 작업으로 넘어간다. 여러 기능/모듈을 동시에 벌리지 않는다.
   - **개발(제품 코드 작성/수정)은 백그라운드 자동 워크플로우·멀티에이전트 하네스로 돌리지 않는다.** 내가 직접, 포그라운드에서, 사용자가 보는 앞에서 한 단계씩 한다. (writer→verifier가 순차라도 금지 — 사용자가 명시적으로 거부함.)
   - 예외: 읽기전용 분석/리서치(웹검색, 코드 탐색)의 병렬 에이전트만 허용. 이건 "개발"이 아님.

2. **최초 정의된 scope에서 확장 금지.** 현재 작업에 정의된 범위 밖의 것은 손대지 않는다. "이왕 하는 김에"식 추가 금지.

3. **큰 줄기 먼저, 잔가지 나중.** 핵심 경로(트렁크)를 완전히 끝낸 뒤에 엣지케이스/세부를 처리한다.

4. **눈속임 금지.** "지금의 오류를 탈피하려 지금만 작동하는 해결책"(테스트 통과용 하드코딩, fixture 값 특수처리, 케이스별 특수분기, band-aid)을 **개발도 수정도 하지 않는다.** **항상 원인부터** 잡는다.
   - 단, 원인 수정이 scope 확장으로 뻗지 않게 항상 경계한다.
   - **원인이 scope/계약(스키마 등)을 건드려야만 해결되면 → 멈추고 사용자에게 flag한다.** 몰래 확장하지도, 하드코딩으로 우회하지도 않는다.

5. **작업 전 이 파일 확인.** 진입 전 위 규칙을 점검하고, 충돌이 보이면 사용자에게 고지한다.

## 프로젝트 현재 상태 (오리엔테이션)

- **무엇:** 도메인 비종속 **의미 기반 Fallback 할당 엔진**(Core). 유행 숏폼의 편집 문법을 사용자 실촬영본으로 재현. 포폴용.
- **포지셔닝:** 생성형(Bluma 등)이 아니라 **실촬영본 채우기**(2026 AI영상 반감/진정성 프리미엄 데이터 근거). 검증 스코프는 국내 푸드 쇼츠로 한정, 아키텍처는 도메인 비종속.
- **방법론:** 하네스 우선. 오라클(불변식+합성 forced 케이스+config 노브)을 먼저 세우고, 엔진을 그 green/red 신호에 대고 test-first로 수렴. 단단함 등급은 [decisions/schema_contradictions.md](decisions/schema_contradictions.md) 참조.
- **진행:** STEP 1✅ → STEP 2(오라클)✅ → STEP 3(Agent A: 3축 스코어링/RULE6, Soft Penalty/RULE10)✅ → STEP 3c(Agent B: 1-Hit 비트 검증/RULE12, 서사모드 bypass)✅ `12 passed`. **다음:** Fallback 강제할당(RULE11/12) — Agent B 반려 시 오디오 리듬 기준 강제. 그 다음 Yellow 추출. (한 번에 하나.)
- **스택/스코프 경계:** Python/pydantic. 플랫폼=YouTube Shorts 전용(IG/TikTok 안티봇 제외). 스코프 밖(현재 미구현·건드리지 말 것): speed_ramp, Cascading Shift, RULE11 슬라이더 실모델 보정, 실제 임베딩/VLM 모델, Yellow/Blue/Pink 파이프라인.
